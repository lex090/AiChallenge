# DDD Audit Checklist

When adding, modifying, or extending domain entities, aggregates, value objects, repositories, or services — validate the change against the rules below. Every decision (good or bad) must cite a specific rule, not subjective preference.

---

## When to Apply This Checklist

- Adding a new Entity, Value Object, or Aggregate
- Modifying aggregate boundaries (moving entities between aggregates)
- Adding/changing repository interfaces
- Adding/changing domain services or use cases
- Adding/changing domain events
- Changing invariants or business rules

---

## Evans (2003) — Strategic & Tactical Patterns

### E1. Bounded Context

> "A Bounded Context delimits the applicability of a particular model. Within a context, the model is kept strictly consistent."

- Each BC has its own Ubiquitous Language — same term must not mean different things within one BC.
- Boundaries must be explicit (module-level separation).
- Shared Kernel between contexts must be minimal (only ID types and shared VOs).

### E2. Entity

> "An object defined primarily by its identity, rather than its attributes."

- Entity has a unique identity that persists through its lifecycle.
- Two entities with identical attributes but different IDs are different objects.
- Identity must be generated by the domain (not by the database).

### E3. Value Object

> "An object that describes some characteristic but carries no concept of identity."

- Immutable. No setters, no mutation methods.
- Equality by attributes (structural equality).
- Side-effect free operations (return new instance).
- Replaceable — swapping one VO for another with same attributes has no consequence.
- Database surrogate keys for VOs are allowed (persistence concern, not domain).

### E4. Aggregate

> "Cluster the Entities and Value Objects into Aggregates and define boundaries around each. Choose one Entity to be the root."

- Root Entity has global identity.
- External objects reference only the root (or hold ID references to internals for lookup).
- Delete root = cascade delete everything inside.
- Invariants of the aggregate are checked on every mutation.
- One transaction per aggregate modification.

### E5. Repository

> "For each type of Aggregate, create a Repository. It should act as an in-memory collection."

- One Repository per Aggregate Root.
- Interface mimics collection: add, get, remove, list.
- Repositories for Value Objects are acceptable only when VOs have independent lifecycle and live in a separate Bounded Context.

### E6. Domain Service

> "When an operation doesn't belong to any Entity or Value Object, model it as a Service."

- Stateless.
- Interface defined in terms of domain model.
- Operation name is part of Ubiquitous Language.

### E7. Factory

> "When creation of an Aggregate is complex, use a Factory."

- Only introduce when construction logic is non-trivial.
- Do not create Factories for simple constructors.

## Vernon (2013) — Aggregate Design Rules

### V1. Protect True Invariants Within Consistency Boundaries

> "A properly designed Aggregate can be modified with its invariants completely consistent within a single transaction."

- **Test:** "If this rule is temporarily violated, is the system in a corrupt/incorrect state from a business perspective?"
- If yes — it's a true invariant, must be in the same aggregate.
- If no — eventual consistency is acceptable (use Domain Events).

### V2. Design Small Aggregates

> "Prefer small Aggregates. Ideally: Root Entity + Value Objects only."

- Only include child Entities if they participate in invariants of the root.
- **Test:** "Is this child entity inside the aggregate because of a true invariant, or just for navigational convenience?"
- If navigational convenience — reference by ID, make it a separate aggregate.
- Large aggregates cause contention (concurrent modification conflicts) on a server.

### V3. Reference Other Aggregates by Identity Only

> "Use only identity references to other Aggregates."

- No object references between aggregates in persistent state.
- Domain Events may carry data snapshots (especially for immutable entities).

### V4. Use Eventual Consistency Outside the Boundary

> "When a command on one Aggregate leads to rules being fulfilled on others, use eventual consistency."

- Mechanism: Domain Events.
- **Test:** "If the other aggregate is updated milliseconds later, does the user observe incorrect behavior?"
- If yes — consider synchronous update in Application Service (pragmatic compromise, two aggregates in one use-case transaction).
- If no — use Domain Event + handler.

### V5. Application Service vs Domain Service

> "Application Services coordinate. Domain Services contain business logic."

- **Application Service (Use Case):** orchestration, transaction boundaries, event publishing. Contains NO business logic.
- **Domain Service:** business logic that doesn't belong to any entity/VO. No infrastructure dependencies.
- Do not mix orchestration with domain logic in one class.

### V6. Domain Events

> "Use Domain Events to capture occurrences. Names in past tense. Immutable."

- Event names: past tense (TurnRecorded, SessionCreated).
- Events are immutable data classes.
- Publish AFTER aggregate state is committed.
- Events carry enough data for subscribers (avoid forcing subscribers to query back).

### V7. Repository — Persist Whole Aggregate

> "The Repository should store the entire Aggregate in one call and reconstitute it in one call."

- If the aggregate is too large to load fully — either:
  - Make the aggregate smaller (V2), or
  - Document lazy loading as a conscious trade-off (ADR).

## Audit Output Format

For each domain change, document:

```
### [Entity/VO/Aggregate/Service Name]

**Change:** What is being added/modified.

**Rules applied:**
- [Rule ID]: [How this rule applies] → [Compliant / Violation / Conscious trade-off]

**Decision:** [What was decided and why]
```
